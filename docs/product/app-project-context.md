## 1. 项目定位

仓库：

```text
x0815fm-cpu/ai-ip-content-app
```

产品不是普通 AI 写作工具，而是：

> 帮助普通人发现自己的真实经历，持续表达人格魅力，最终建立长期个人 IP。

第一批目标用户：

> 拆书会里想通过短视频或图文表达自己、吸引同频用户的会员。

第一阶段业务闭环：

```text
真实表达
→ 吸引同频用户
→ 带入拆书会
→ 获得佣金
→ 建立信心
→ 持续表达
→ 长期 IP
```

---

## 2. 产品核心原则

### 2.1 长期 IP 的核心

长期 IP 的核心不是单一知识标签，而是人格魅力。

人格魅力来自：

- 真实经历
- 内心困境
- 认知变化
- 价值观
- 专业能力
- 面对失败时的选择
- 持续成长的过程

### 2.2 第一条内容的入口

第一条内容优先从：

> 一次真实的内心失败

开始。

失败不是为了卖惨，而是为了寻找：

```text
失败前相信什么
→ 失败时经历什么
→ 后来是否发生变化
→ 今天如何理解这件事
```

真实性底线：

> 用户没有讲出的时间、人物、事件、选择、结果、变化和新信念，AI 不得编造。

### 2.3 转折点定义

完整转折点是：

```text
低谷 + 选择 + 变化 + 新的信念 = 转折点
```

用户输入的失败故事只是原始素材，字段统一使用：

```ts
failureStoryInput
```

不要把原始输入直接命名为转折点。

## 3. 用户当前流程

```text
首页
→ 三道选择题
→ 推荐三个方向
→ 用户选择方向
→ 输入一次真实失败或跳过
→ 推荐三个选题
→ 用户选择选题
→ 生成内容文案
→ 反馈改写
→ 沉淀故事
```

目前真实模型链路已经较完整的部分：

```text
选择题
→ 方向推荐
→ 真实失败输入
→ 选题推荐
```

文案生成、反馈改写和故事沉淀仍需要继续完成真实模型闭环。


最终验收文档：

```text

```

完成内容：

- 模型顶层输出使用 `{ "topics": [...] }`。
- Provider 对 `recommend_topic` 进行任务专用结构校验。
- 校验通过后解包为 `TopicRecommendation[]`。
- 无效输出触发现有 mock fallback。
- Model Service Contract 增加 `failureStoryInput`。

PR #8 的重要经验：

空故事时，模型曾虚构“入职第一周”“被否定的汇报”“职场第一年”等经历。经过三轮真实模型测试，最终明确：

- 无故事时不能创造具体场景。
- 不得补充未提供的时间、人物、事件、结果和变化。
- 选题使用动机式、当前式、开放式表达。
- 标题中的事实必须能够从 payload 溯源。

最终通过的空故事选题示例：

```text
我为什么想记录职场成长
工作里，我现在最想改变什么
对我来说，记录成长意味着什么
```

## 5. 当前真实状态

### 5.1 ContentScreen 已有防白屏保护

`main` 中已经包含：

```ts
const rows = content?.rows ?? [];
const publishHint = content?.publishHint ?? "";
```

当 `rows` 为空时，页面显示：

```text
内容生成遇到问题，请返回重试。
```

当 `publishHint` 缺失时，不渲染发布建议区域。

这解决的是“页面不要白屏”的保险问题，不代表真实文案生成已经完成。

### 5.2 generate_content 根因仍未完成

当前仍需重点核对并完成：

- `generate_content` 是否仍落入 `DEFAULT_PROMPT`。
- DeepSeek Provider 是否尚未校验 `generate_content` 返回结构。
- 是否已存在正式 `prompts/generate_content.md`。
- 是否已完成真实 DeepSeek 文案生成验收。

仅有 ContentScreen 防御代码，不足以宣布第一条内容体验完成。

### 5.3 API 响应结构

`/api/generate` 的标准响应是：

```json
{
  "ok": true,
  "data": {},
  "source": "deepseek",
  "error": ""
}
```

`modelService` 读取的是 `result.data`。

测试时若直接模拟：

```json
{
  "content": {}
}
```

前端会认为响应无效并 fallback 到 mock，无法真正测试异常数据分支。

真实验收必须检查：

```text
source === "deepseek"
```

若为 `source === "mock"`，即使页面正常，也只能说明 fallback 正常，不能说明真实模型通过。

## 7. 下一轮 PR 的核心要求

建议名称：

```text
PR-09：First Content Experience V1
```

必须包含：

### 7.1 Generate Content Prompt

输入：

```text
answers
selectedDirection
selectedTopic
failureStoryInput
```

有失败故事时：

- 只使用真实信息。
- 不编造完整转折。
- 可保留未解决状态。

空故事时：

- 不虚构经历。
- 围绕方向、目标、动机和当前思考。
- 若选题包含未经证实的具体经历，必须改写为开放式、动机式或当前式表达。

### 7.2 模型输出结构

```json
{
  "content": {
    "rows": [
      { "label": "标题", "value": "..." },
      { "label": "开头", "value": "..." },
      { "label": "正文", "value": "..." },
      { "label": "结尾", "value": "..." }
    ],
    "publishHint": "..."
  }
}
```

Provider 校验后解包为当前前端使用的 `GeneratedContent`。

### 7.3 任务专用结构校验

仅在：

```ts
taskType === "generate_content"
```

时执行。

至少验证：

- 顶层是对象。
- 存在 `content` 对象。
- `rows` 是非空数组。
- 每个 row 的 `label`、`value` 是非空字符串。
- 至少包含标题、开头、正文、结尾。
- `publishHint` 是非空字符串。
- 无效则返回 `ok: false`，触发现有 mock fallback。

### 7.4 用户可体验状态

用户点击生成后显示：

```text
小光正在整理你的第一条内容……
```

真实模型失败时：

- 页面不白屏。
- 自动显示备用内容。
- 开发验收可以通过 API 响应查看 `source` 和 `error`。

### 7.5 必须完成的验收

- 有 `failureStoryInput`。
- 空 `failureStoryInput`。
- 非法模型输出。
- 无 API Key。
- 从首页完整走到文案页。

## 9. 重新确立的项目推进计划

### 阶段 1：完整生成第一条内容

用户能够从首页走到真实文案页。

### 阶段 2：反馈改写真正可体验

用户点击“不够像我”“太官方”“太长”等反馈后，文案真实变化，失败时保留原文。

### 阶段 3：故事资产可体验

用户能将内容沉淀为故事卡，并在故事库重新看到。

### 阶段 4：第一次使用体验优化

完善加载、失败、重试、返回、输入保留和手机适配。

### 阶段 5：让用户愿意发布

完善平台适配、口语感、字数、复制、拍摄建议和自然转化。

### 阶段 6：持续表达闭环

第二次回来不从零开始，已有故事能够支持新选题和个人 IP 档案。

### 阶段 7：内容规划与商业闭环

形成周/月内容规划、人格内容与转化内容搭配，以及发布效果复盘。

## 10. 固定研发流程

```text
产品讨论
→ 产品文档
→ Codex 技术方案
→ 技术方案 Review
→ Codex 开发
→ PR 代码 Review
→ 真实模型验收
→ 从首页完整回归
→ 用户亲自体验
→ Merge
```

新增硬门槛：

> 主流程任何一步不能完成，就不能开始下一个 PR。

每一轮必须回答：

1. 用户在哪个页面看到变化？
2. 用户可以亲手完成什么新操作？
3. 用户是否能从首页完整走到目标页面？
4. 结果是否由真实模型产生，而不是 mock 掩盖？
5. 用户是否实际感受到了改进？

纯后台改动不能单独算一轮产品成果。

## 11. 协作要求

ChatGPT 在项目中承担：

- 产品负责人
- 产品文档负责人
- 技术方案 Review
- PR Review
- 全链路体验验收负责人

协作规则：

1. 一次只推进一个可体验目标。
2. 不因代码、文档或 Prompt 完成就宣布产品完成。
3. 每个 PR 后必须完整走一次网页流程。
4. 每次给 Codex 指令时自动附上一键复制话术。
5. 用户无需再次提醒“一键复制”。
6. 重要结论更新仓库，但不以文档数量代替产品进展。
7. 所有改动最终必须让用户看得见、点得到、感受得到。
8. 用户指出流程乱时，先停下来复盘。
9. 不重复询问已经确定的产品信息。
10. 所有真实模型验收必须检查 `source`。

## 12. 新对话框启动检查

新对话框开始时，不要立即扩功能。先以 `main` 为准核对：

1. `ContentScreen` 防白屏代码是否仍存在。
2. `generate_content` 是否仍使用 `DEFAULT_PROMPT`。
3. Provider 是否已经校验 `generate_content` 返回结构。
4. 是否已有 PR-09 技术方案或功能分支。
5. 当前从首页到文案页的真实表现。
6. 当前唯一阻塞用户体验的问题。

然后只输出：

- 当前真实状态
- 下一步唯一目标
- 完成后用户能体验到什么
- 一键复制给 Codex 的话术

北极星：

> 先让用户完整用起来，再让模型越来越聪明。
