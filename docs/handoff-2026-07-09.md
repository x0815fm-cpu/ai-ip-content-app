## 三、当前架构

```
前端页面
  ↓
modelService.generate(taskType, payload)  ← 前端统一入口
  ↓
POST /api/generate                        ← 服务端 API Route
  ↓
deepseekProvider / mockProvider           ← 真实模型 or mock fallback
  ↓
promptRegistry.getSystemPrompt(taskType)  ← Prompt 管理
  ↓
结构化 JSON 返回
  ↓
页面展示
```

三层 fallback：
1. 无 API Key 或 `USE_MOCK_MODEL=true` → 服务端直接返回 mock
2. DeepSeek 请求失败 → 服务端 fallback 到 mock
3. `/api/generate` 网络失败 → 客户端动态 import mock

## 六、踩过的坑

### 6.1 Git 网络超时
本地 `git fetch` 经常超时。解决：优先用 GitHub MCP API（push_files / create_branch / Git Data API）操作远程，本地只做验证。

### 6.2 Squash Merge 后分支不兼容
PR #1 用 squash merge 后，原来的 feature 分支和新 main 历史完全不同，导致后续 PR 合并冲突。解决：每次新 PR 都从最新 main 的 SHA 用 `create_branch` 创建全新分支。

### 6.3 GitHub Token 权限
Fine-grained Token 初始没有任何权限。需要在 Settings 里手动添加 Contents + Pull requests 的 Read and write。Draft→Ready 操作需要 GraphQL mutation（REST API 不支持）。

### 6.4 push_files 大内容报错
MCP push_files 推送大文件内容时可能触发参数验证错误。解决：fallback 到本地 git commit + push。

### 6.5 ImageGen 无透明背景
图片生成工具输出 RGB 模式（无 alpha 通道），即使 prompt 要求透明背景。解决：后处理用 PIL 加圆形 mask + alpha 通道。

### 6.6 npm run dev 修改 next-env.d.ts
Next.js dev server 会自动往 next-env.d.ts 加 import 行。解决：提交前必须清理该文件。
