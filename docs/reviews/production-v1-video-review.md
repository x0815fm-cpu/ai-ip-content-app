# Production V1 真机录屏评审

日期：2026-07-11  
依据：用户在 iPhone Safari 中完成约 35 秒真实操作录屏。

## 结论

主流程已经跑通，但当前线上版本仍明显像演示原型，不适合继续扩功能。

核心问题不是“功能缺失很多”，而是：

1. 真机适配不成立。
2. 内容链路前后不一致。
3. 页面存在虚假或误导性的产品状态。
4. 最终结果缺少真正可用的发布动作。

## P0 问题

### 1. 手机里又套了一层假手机

真机 Safari 顶部已经有系统状态栏，App 内又渲染了 `9:41` 假状态栏和 phone shell，造成：

- 双状态栏。
- 页面左右留白，像原型预览而不是正式 App。
- Safari 底部工具栏遮挡方向页按钮。
- 小光悬浮球持续压住按钮或内容。

需要在移动端使用真实全屏布局：隐藏模拟状态栏和手机外壳，处理 `safe-area-inset-*`，保证底部操作始终可见。

### 2. 用户选择的选题没有被最终文案遵守

录屏中用户选择的是：

`这一路上我最大的一个改变`

最终文案标题却变成：

`我为什么想开始做这件事`

这说明“选题 → 文案”没有形成强约束。用户做了选择，但结果像回到了第一个默认选题。

### 3. 跳过失败故事后，选题仍在暗示不存在的经历

用户选择“暂时跳过”后，系统仍推荐：

- `这一路上我最大的一个改变`
- `如果重来一次，我会早点明白这件事`

这些标题默认用户已经经历过“改变”或“重来”，与“未提供真实故事时不得虚构经历”的原则冲突。

### 4. 反馈改写结果不可直接发布

用户点击“太营销了”后，改写内容直接出现：

`按“太营销了”改了一版……`

这属于系统解释，不是用户可发布的正文。反馈词不应进入成稿，改写后应只返回新的正文。

### 5. 故事库展示了假的个人资产

故事库默认展示：

- 2023/2024 的日期。
- 多条看似属于用户的故事。
- 收藏数字。

这些内容并非用户真实创建，却以“我的故事库”呈现，会严重损害信任。

用户点击“沉淀到故事库”后，只是返回同一批预置内容，没有看到刚生成的内容被新增，按钮行为与承诺不一致。

处理原则：

- 未实现真实保存前，故事库应显示真实空状态或明确标注“示例”。
- “沉淀到故事库”必须真的保存并可见；否则暂时改成不承诺保存的动作。

## P1 问题

### 6. 方向与选题卡片信息过长

三个方向和三个选题都包含多段解释，手机上阅读成本很高，用户难以快速比较。

应优先显示：标题 + 一句话差异；详细原因按需展开。

### 7. 首页视觉仍有原型感

首页主视觉顶部存在一行模糊、重复的说明文字，信息层级不清；整体像设计稿嵌入网页。

### 8. 最终文案页缺少核心动作

页面写着“可以直接发布”，但没有明显的：

- 一键复制全文。
- 分段复制。
- 分享或导出。

生成结果无法顺手带走，闭环不完整。

### 9. 加载按钮视觉仍像可点击

生成中虽然已禁用，但颜色和按钮形态几乎不变，只替换了文字，用户不容易感知“正在处理且不可重复点击”。

## 下一轮唯一目标

`Production V1 P0 Trust Fix`

目标不是新增功能，而是让当前线上主流程从“演示”变成“可信可用”。

本轮优先完成：

1. 真机全屏与底部安全区适配。
2. 强制最终文案严格遵守已选选题。
3. 跳过故事时不得生成暗示既有经历的选题。
4. 反馈改写不得把反馈指令写进正文。
5. 移除或明确标注故事库假数据；禁止假保存。

暂不做数据库、长期记忆、复杂 Agent 或更多功能页面。

## 网页验收标准

在 iPhone Safari 上实际完成一次全流程：

- 页面没有双状态栏和手机壳。
- 所有底部按钮不被 Safari 工具栏或悬浮球遮挡。
- 选择哪个选题，最终文案就围绕哪个选题。
- 跳过故事时，不出现未经用户提供的过去经历。
- 点击反馈后，只出现可发布的新文案，不出现系统解释。
- 故事库不再冒充用户已有真实资产。
