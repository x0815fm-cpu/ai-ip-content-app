# Review 008：PR-08 真实模型验收复核

日期：2026-07-10  
评审对象：PR #8 `feat: implement recommend_topic prompt v2 with validation`  
依据：
- `docs/reviews/review-006-pr-08-technical-plan.md`

## 结论

**场景一通过；场景二未通过。PR #8 暂不合并。**

## 场景一：有 failureStoryInput

输入中包含疲惫、工作做不好、不敢告诉别人、自我怀疑。

返回选题围绕这些真实信息展开，没有补写“已经走出低谷”“形成新信念”等用户未提供事实。

结论：通过。

## 场景二：failureStoryInput 为空

已知输入只有：

- 素材来源：工作/职业
- 内容目标：记录成长
- 发布载体：小红书/图文
- 内容方向：职场成长实录
- failureStoryInput：空字符串

模型返回：

1. `入职第一周，我差点哭出来`
2. `一次被否定的汇报让我学会的事`
3. `我为什么开始记录工作日常`

其中前两条断言了用户并未提供的具体经历：

- 入职第一周差点哭；
- 汇报被否定；
- 从该事件中学会了某件事。

这违反 Review-006 的明确要求：

> 无失败故事时，不得假设用户拥有未提供的经历。

因此，不能把“具体、第一人称”理解成“替用户编造一个具体事件”。具体性必须来自已知信息，或使用不声称事实已经发生的开放式表达。

## 修复要求

仅调整 `recommend_topic` Prompt 设计稿和运行时 Prompt，不需要修改 Provider 校验。

无故事场景增加以下硬约束：

1. 不得生成断言某个具体事件已经发生的标题，除非该事件存在于 payload。
2. 不得使用未提供的时间、人物、场景、失败、评价、结果或变化。
3. 选题应锚定已经提供的信息，例如“工作/职业”“记录成长”“职场成长实录”。
4. 可以使用开放式、反思式或动机式标题，但不能伪造回忆。

可接受方向示例：

- `我为什么想开始记录工作日常`
- `工作里最让我想改变的状态`
- `我想用内容记录怎样的职场成长`

不可接受示例：

- `入职第一周，我差点哭出来`
- `一次被否定的汇报让我学会的事`
- `被领导批评后，我终于明白了什么`

## 重新验收标准

完成 Prompt 修订后，使用同一组空故事输入重新真实调用 DeepSeek，并确认：

- `source === "deepseek"`；
- 恰好 3 个选题；
- 恰好 1 个推荐项；
- 三个选题具体但不虚构经历；
- 每个标题都可以从现有 payload 直接得到依据。

通过后再进行最终合并审核。
